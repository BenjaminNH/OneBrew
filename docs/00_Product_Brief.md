# Product Brief: OneBrew

## 1. Core Problem
目前市面上的手冲咖啡评价与记录应用存在明显的两极分化与体验痛点：
1. **生态与平台空缺**：优秀的记录应用（如《咖啡札记》）多为 iOS 独占，庞大的安卓市场缺乏优质的纯粹工具。
2. **体验极其臃肿且吃相难看**：部分全能型 App 被硬件生态绑架（如《享咖小记》），UI 呈生硬的表单堆砌态势，且为了卖硬件而对免费功能（如可添加的器具数量）进行恶意限制。
3. **极高的使用门槛与强制流程**：一上来就要求填写多维度的 SCAA 风味图表，或者在冲煮前必须先去建库录入咖啡豆详情和磨豆机型号。导致冲煮这件原本享受的事情变成了“上班打卡”。

用户核心需要的是一个能够**优雅、丝滑、无压迫感**地记录冲煮参数（如研磨度、粉水比、时间等）并反馈风味（好喝/不好喝，或具体的酸甜感）的工具，以便于未来重现这杯咖啡的好味道。

## 2. The User & Context
*   **目标人群**：手冲咖啡与意式咖啡爱好者。既包括每天“只求快速冲一杯好喝无雷咖啡以便去上班”的入门者，也包括“喜欢死磕水温和研磨度微调”的硬核极客玩家。
*   **使用场景**：用户通常在吧台前、准备冲煮时或刚刚喝完咖啡时使用。此时用户可能满手咖啡粉、拿着手冲壶或处于放松状态。在这一场景下，UI 交互必须**极尽顺滑、秒开即用、且有“肌肉记忆”般的填表体验**。

## 3. The MVP (Minimal Viable Product)
*   **技术栈选型**：跨平台框架 **Flutter**（初期主攻安卓市场抢占空位，后期兼顾 iOS），采用 **Local-First (纯本地离线优先)** 存储策略（如 SQLite/Isar 数据库），初期无需云端服务器。
*   **核心功能模块**：
    1.  **极简心流记录器 (Quick Logger)**：大入口，秒开计时器。采用“渐进式入参”（默认只展示必备参数如：豆种、粉水比、总用时；点击展开后才有水质、室温、三段式等多维参数）。
    2.  **无摩擦力录入机制 (Frictionless Inventory)**：不强制提前创建咖啡豆或设备的数据库。用户只需手打一次标签（如“C40”，“花魁”），系统自动记住并成为下拉候选；同时提供“复制昨日参数/再冲一次”的快捷模板。
    3.  **分层风味评价体系 (Tiered Rating System)**：对入门小白提供直觉化的滑动星级/表情好评；为核心玩家提供折叠在底部的风味轮、酸甜苦分离滑块。
    4.  **本地数据墙与历史过滤**：展示既往的冲泡记录，高分冲煮高亮显示，建立用户的冲煮成就感。

## 4. The OKRs
**Objective 1: 完成“纯净、极致优雅”MVP 版本的开发，并实现代码级的自动化自测验证**
*   **KR 1**: 完成基于 Isar/SQLite 的本地数据库建模，并实现 BrewRecord, Bean, Equipment 增删改查逻辑的自动化单元测试，且核心查询（如无感复用数据提取）的测试覆盖率达到 90% 以上。
*   **KR 2**: 完成 Flutter 端 UI 核心交互（如秒开新建、倒计时/正计时机制、历史记录流）的代码开发，通过 Widget Tests 确保页面跳转无异常、关键 UI 组件（如开始按钮、评价滑动条）断言渲染成功率达 100%。
*   **KR 3**: 跑通自动化构建验证流程（CI/CD 或脚本），成功打包出无编译 Error、无明显内存泄漏的可用 Android APK 安装包。

**Objective 2: 建立技术壁垒与可验证的 UX 性能指标**
*   **KR 1**: 构建应用冷启动到计时器页面的集成测试用例，通过自动化脚本当面验算：完成从“启动应用”到“触发开始冲煮计时”之间的 UI 节点点击次数必须 `<= 2` 次。
*   **KR 2**: 针对“极简模式与专业模式的无缝展开”及“后台挂起时应用计时异常”等情况编写并跑通特定的边界测试与集成测试（Integration Tests），要求 0 Error 修复完成。
